# EMAIL_PHASE.md - Tallyo

Implementation plan for adding Resend email to Tallyo in safe stages.

## Current Scope

Build transactional email first. Do not start automated reminders until manual invoice sending, event logging, and failure handling work reliably.

- Done: Resend sending domain `mail.tallyo.co.uk`.
- Done: manual `send-document-email` Edge Function.
- Done: in-app Email action for saved documents.
- Done: `audit_events` table applied.
- Done: `resend-webhook` Edge Function receiving signed Resend events.
- Verified: `document_email_sent`, `email_sent`, and `email_delivered` audit events are recorded with document/user tags.
- Done: delivery status appears in the invoice list and document activity panel.
- Resend webhook currently listens for: `email.sent`, `email.delivered`, `email.delivery_delayed`, `email.bounced`, `email.complained`, `email.clicked`, `email.failed`, `email.opened`, and `email.received`.
- In progress: opt-in automatic email for invoices generated by recurring schedules.
- Next: deploy and test recurring auto-email, then plan overdue reminders.

## Stage 1 - Resend And DNS Setup

User/provider work:

1. Create or open the Resend account.
2. Add a sending domain. Selected for Tallyo: `mail.tallyo.co.uk`.
3. Add the DNS records Resend provides for SPF and DKIM.
4. Add a starter DMARC TXT record:

   ```text
   Name: _dmarc
   Type: TXT
   Value: v=DMARC1; p=none;
   ```

   Once delivery is confirmed and all legitimate senders pass DMARC, tighten later to `quarantine`, then eventually `reject`.

5. Verify the domain in Resend.
6. Create a Resend API key.
7. Store the key only as a Supabase Edge Function secret named `RESEND_API_KEY`.
8. Use the initial sender address `Tallyo <invoices@mail.tallyo.co.uk>`.

Do not put the Resend API key in `index.html`, `config.js`, `.env.example`, docs, screenshots, or chat.

## Stage 2 - Manual Send Invoice Function

Build one Edge Function, `send-document-email`.

Function responsibilities:

- Require a logged-in Supabase user JWT.
- Read the requested invoice/quote/credit note from Supabase.
- Verify the row belongs to the authenticated user.
- Build a plain, professional email body server-side.
- Send via Resend using `RESEND_API_KEY`.
- Append a normal document `history` event.
- Insert a trusted `audit_events` row if `supabase/audit_events.sql` has been applied.
- Return success/failure to the browser without exposing secrets.

Browser responsibilities:

- Let the user choose/confirm recipient email.
- Call the Edge Function with the document ID and recipient.
- Show a clear success or failure message.
- Never send the Resend API key or trusted event data.

## Stage 3 - Delivery And Failure Webhooks

Build a second Edge Function, `resend-webhook`.

Webhook responsibilities:

- Accept Resend webhook POSTs.
- Verify webhook signatures with `RESEND_WEBHOOK_SECRET` once configured.
- Store events idempotently by Resend event/email ID.
- Store signed Resend events for `email.sent`, `email.delivered`, `email.delivery_delayed`, `email.bounced`, `email.complained`, `email.clicked`, `email.failed`, `email.opened`, and `email.received`.
- Log delivered, delayed, bounced, failed, or complained events into document history where relevant.
- Never mark a document as sent/delivered from an unverified webhook.

Implementation status:

- `resend-webhook` exists and is deployed.
- Resend webhook URL: `https://cuagwifetheefftleeup.supabase.co/functions/v1/resend-webhook`.
- Events confirmed in `audit_events`: `email_sent`, `email_delivered`.
- Configured Resend events: `email.sent`, `email.delivered`, `email.delivery_delayed`, `email.bounced`, `email.complained`, `email.clicked`, `email.failed`, `email.opened`, `email.received`.
- Delivery/failure events are also eligible to write lightweight invoice history entries for user-visible status.
- The app reads `audit_events` and shows email status badges such as Email sent, Delivered, Delayed, Bounced, Failed, Complaint, Opened, Clicked, or Received.

## Stage 4 - Automated Sending

Manual send + webhook events are stable. Automated sending is being added in controlled, opt-in pieces:

- Done in code: `recurring_templates.email_enabled` flag, off by default.
- Done in code: recurring schedule UI toggle for automatic email.
- Done in code: `generate-recurring` sends generated invoices through Resend only when `email_enabled = true` and the customer has a valid email address.
- Done in code: automatic sends insert `document_email_sent`; signed Resend webhooks continue tracking delivery, bounce, complaint, failed, opened, clicked, and received events.
- Required before live use: run `supabase/recurring_email_enabled.sql`, deploy `generate-recurring`, and test with one schedule.
- Add overdue reminder automation.
- Add suppression rules for paid/cancelled invoices.
- Add reminder cadence rules so customers are not spammed.

## Acceptance Tests

Manual send:

- A signed-in user can send their own invoice to a confirmed email address.
- Another user's invoice cannot be sent.
- Missing recipient is rejected.
- Invalid email is rejected.
- Resend failure does not create a false sent history event.
- Success creates a visible document history entry.
- No secret appears in browser files, logs, or network responses.

Webhook:

- Valid delivery/failure event is stored once.
- Duplicate event is ignored or safely upserted.
- Fake/unsigned request is rejected.
- Event maps back to the correct document and user.

Automation:

- Existing schedules remain manual unless `email_enabled` is turned on.
- Due recurring invoice can generate once and send once when opted in.
- Missing or invalid customer email creates the invoice and records that email was skipped.
- Resend failure creates the invoice and records that email failed; it must not falsely claim delivery.
- Overdue reminder cadence is respected.
